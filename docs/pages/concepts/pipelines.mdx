In Bauplan, pipelines are implicitly defined by chaining models together
through their declared inputs and outputs. Pipelines take the form of
Directed Acyclic Graphs (DAGs).

You don't need to define a DAG manually, because Bauplan automatically
analyzes the dependency graph based on model declarations and
automatically infers the correct execution order.

To chain models into a DAG, simply pass a previous bauplan model as an
input.

```python
@bauplan.model()
@bauplan.python('3.11')
def step_1(data=bauplan.Model('input_table')):
    ...

@bauplan.model()
@bauplan.python('3.11')
def step_2(data=bauplan.Model('step_1')):
    ...

@bauplan.model()
@bauplan.python('3.11')
def step_3(data=bauplan.Model('step_2')):
    ...
```

## Parameterize DAGs with parameters and secrets

In production, it is common for the same DAG code to be run on different days, 
or to call an external service (e.g., an LLM for data enrichment). 
Bauplan supports these scenarios with parameters and secrets.


You can define a parameter using the CLI:

```sh
bauplan parameter set prompt_summary "Write a concise, incisive summary of the news article." --description "Prompt passed to GPT for article summarization"
```

Each parameter is defined in `bauplan_project.yml` with a `type` and
a `default` value.

```yaml
parameters:
  interest_rate:
    default: 3.5
    type: float
  loan_amount:
    default: 200000
    type: int
```

Parameters can be:

-   **Strings** (`str`)
-   **Numbers** (`int`, `float`)
-   **Booleans** (`bool`)
-   [Secrets](/concepts/pipelines#secrets) (encrypted API keys or credentials)

### Parameters

Parameters let you pass runtime values to models without changing code. They can be accessed in a variety of ways.

Parameters can be passed in [`bauplan.Model`](/reference/bauplan#bauplan_model) as a filter,
using the `$param_name` syntax:

```python
@bauplan.model()
@bauplan.python('3.11')
def qualifying_loans(
    data=bauplan.Model(
        'loans',
        filter='interest_rate <= $interest_rate AND amount >= $loan_amount',
    ),
):
    ...
```

<Note>
To see how to use parameters in a model as a variable, see [reference](/reference/bauplan#bauplan_parameter).
</Note>

In SQL, use `$param_name` directly in your query:

```sql
SELECT * FROM loans
WHERE interest_rate <= $interest_rate
  AND amount >= $loan_amount
```

Defaults can be overridden at runtime:

<Tabs queryString="interface">
  <TabItem value="cli" label="CLI">

```sh
bauplan run --param interest_rate=4.5 --param loan_amount=500000
```

  </TabItem>
  <TabItem value="sdk" label="SDK">

```python
import bauplan

client = bauplan.Client()
client.run(
    project_dir="./pipeline",
    parameters={"interest_rate": 4.5, "loan_amount": 500000},
)
```

  </TabItem>
</Tabs>

### Secrets

Secrets are encrypted parameters for sensitive values like API keys,
database credentials, or tokens. They are encrypted, versioned, and
injected into your pipeline code at runtime вҖ” avoiding hardcoding.

Use the CLI with `--type secret` to create one:

```sh
bauplan parameter set openai_api_key sk-abc123... --type secret
```

Bauplan encrypts the value via AWS KMS and stores only the ciphertext
in your `bauplan_project.yml`. Secrets are never stored or transmitted
in plaintext.

In your code, reference secrets with `bauplan.Parameter` just like
regular parameters.

For a complete walkthrough using secrets with an LLM pipeline, see the
[Using LLMs with Secrets](https://github.com/BauplanLabs/examples/tree/main/learn/04-using-llms-with-secrets)
example.

## Best Practices

To keep your project organized and make it easy to run, test, and
inspect individual pipelines, we recommend the following structure. See
[Projects](/concepts/projects) for more details.

```text
my_project/
в””в”Җв”Җ pipelines/
    в”ңв”Җв”Җ sales_reporting/
    в”Ӯ   в”ңв”Җв”Җ models.py
    в”Ӯ   в””в”Җв”Җ bauplan_project.yml
    в”ңв”Җв”Җ customer_segmentation/
    в”Ӯ   в”ңв”Җв”Җ models.py
    в”Ӯ   в””в”Җв”Җ bauplan_project.yml
    в””в”Җв”Җ churn_prediction/
        в”ңв”Җв”Җ models.py
        в””в”Җв”Җ bauplan_project.yml
```

-   Group models that form a logical pipeline into a single file:
    `models.py`.
-   Place that file inside a folder named after the pipeline
    (`sales_reporting`, `churn_prediction`, etc.).
-   Each pipeline folder contains its own `bauplan_project.yml` file,
    which defines environment settings and dependencies for that
    pipeline only.
-   Separate function bodies into external modules and call them within
    the Bauplan models. This keeps business logic code neatly separated
    from the DAG and environment declaration, making code refactoring
    easier and future-proof.

This layout:

-   Makes each pipeline self-contained and easy to run with
    `bauplan run` from inside the folder.
-   Keeps models modular and promotes reuse.
-   Helps enforce consistent dependency tracking and ref resolution.
